# Match Screen Performance Plan

Grounded in a read-through of the current `MatchScreen` render path, `board/projection`, `matchStore`, and the 24 components under `src/app/components/match/`. Phases 0, 1, and (a scoped) 2 are implemented (see their results sections below); Phases 3–4, Phase 1.5, and Phase 2.1 are still planning-only.

## What's actually causing the sluggishness

1. **`MatchScreen.tsx` (2,501 lines) subscribes to the entire engine state as one blob** — `useMatchStore((s) => s.state)`. The engine returns a brand-new `GameState` object on every valid action (by design — that's the "new state each dispatch" contract). So every single action, no matter how small, produces a new top-level reference and forces this one giant component to re-evaluate in full.

2. **Zero `React.memo` anywhere in the match UI.** Checked all 24 files in `src/app/components/match/` — none of them (`PlayerBoardPanel`, `BoardCardTile`, `DockHand`, `DonStack`, `PileStack`, `ActionBar`, `ActionLogPanel`, `PendingChoicePrompt`, etc.) are memoized. There is no re-render firewall anywhere in the tree, so #1's full-tree re-render actually reconciles the entire board every time, not just the card that changed.

3. **Board projection is rebuilt from scratch on every render, unmemoized.** `MatchScreen.tsx:361-362` calls `projectPlayerBoard(matchState, defs, images, ...)` directly in the render body for both players — no `useMemo`. This walks every zone (hand, characters, stage, life, DON, trash, deck) for both sides and rebuilds `CardView` objects each time, even for renders triggered by something UI-only (opening the log panel, a hover state, a modal).

4. **Card art has no responsive/thumbnail variant.** `cardView.ts` resolves one fixed `imageUrl` per card (`images[instance.cardDefinitionId]`) — the same 84–132KB webp is used whether it's a 150×210px board tile or a full-size zoom preview. No `loading="lazy"`, no explicit `width`/`height` on the `<img>` tags in `BoardCardTile.tsx` (layout-shift risk on top of the transfer weight).

**Verified NOT a problem** (worth not re-investigating later):
- `CardMovementOverlay` uses the Web Animations API (`node.animate`) with correct cleanup (cancels timers/animations on unmount) — GPU-composited, not a likely main-thread cost.
- `ScaleToFit` already went through 3 iterations and landed on pure CSS `container-type: size` + `cqh` units — no JS-computed transform scaling left.
- Zustand stores are already split by concern (`matchStore`, `cardAnimationStore`, `phaseAnnounceStore`, `settingsStore` are separate stores), so the blast radius is contained *within* matchStore, not cross-store noise.

## Phase 0 results (baseline measurement — done)

Two diagnostic test files were added as permanent, re-runnable tools (not one-off scripts):

- `src/board/projection/__tests__/projectPlayerBoardCost.phase0.test.ts` — runs in plain Node, no browser needed. **Ran successfully in this environment.**
- `src/app/screens/__tests__/matchScreenPerf.phase0.test.tsx` — mounts the real `MatchScreen` in jsdom via `React.Profiler` and counts commits + `projectPlayerBoard()` calls per dispatched action (give DON, declare attack, activate blocker). **Written and ready, but could not be executed in this sandbox** — jsdom hangs indefinitely here even for a zero-React sanity test (isolated by testing a trivial `document.body.innerHTML` assertion under `// @vitest-environment jsdom` with no React at all — it hung the same way, ruling out this test's own code as the cause). This is an environment issue with this sandbox, not the test. Run it locally (`npx vitest run src/app/screens/__tests__/matchScreenPerf.phase0.test.tsx`) or in CI to get real commit counts before starting Phase 1.

**What the Node benchmark actually measured** — `projectPlayerBoard()`'s own compute cost, at a representative mid-game board (5 characters/side, 6 hand cards/side, 5 life cards/side, DON on both sides), 2000 iterations:

- ~0.05ms per both-sides call (both players projected, as `MatchScreen.tsx:361-362` does every render) on this sandbox's CPU.
- Scales with board size as expected: 2 characters/side → 0.029ms, 5 → 0.033ms, 10 → 0.048ms (both-sides call).

**This changes the emphasis of Phase 1.** The projection function itself is cheap in isolation — sub-millisecond even completely unmemoized. The actual cost isn't rebuilding the projection data; it's that every one of the ~24 match components has zero `React.memo`, so a cheap-but-unmemoized projection call is followed by React re-rendering (running every component function, then diffing, then committing) the *entire* board tree on every single dispatch, no matter how small the change. `useMemo`-ing the projection call still matters (it's still needless recomputation and it produces new object references that would defeat `React.memo` even if added), but **`React.memo` on the leaf/mid components is where the real win is**, not the projection math. Phase 1 below is unchanged in content but this is why it's ordered "memoize projection AND add React.memo together" rather than treating the projection as the expensive part.

Real render-commit counts, main-thread time under CPU throttling, and low-spec-device FPS still need the jsdom harness (or a real device) — that's Phase 4's job once Phase 1 lands, and remains a known limitation of what could be verified from here today.

## Phase 1 results (re-render blast radius — done, needs local verification before merge)

Scope grew significantly beyond the original "add `useMemo` + wrap 7 components in `React.memo`" plan, discovered mid-implementation:

- **`useBoardSelection.ts` (1,309 lines) had to be touched too.** It returns ~28 functions (`hasActivateMain`, `canGiveDonOnCard`, `handleCardTap`, etc.) and a handful of derived values (`counterProgress`, `donChoiceProgress`, `fieldChoiceInfo`), every one of them a fresh closure/object on every render — none `useCallback`/`useMemo`'d. These flow straight into `PlayerBoardPanel`/`DockHand`/`ActionBar` as props (some directly, some called *during* the child's own render, e.g. `hasActivateMain(card)`). Wrapping those 3 components in `React.memo` without fixing this would add comparison overhead for zero benefit — exactly the failure mode this doc's "Known limitations" section already called out.
- Rather than hand-writing a `useCallback` dependency array for each of the ~28 functions (real risk of a stale-closure correctness bug in the single most gameplay-critical file in the app, with no way to verify by rendering here — jsdom is broken in this sandbox), added `src/app/hooks/useStableDelegates.ts`: a small generic "latest ref" hook. Every function gets a permanently-stable wrapper that always delegates to the current render's real implementation via a ref — standard, low-risk pattern for exactly this situation, no per-function dependency bookkeeping. `counterProgress`/`donChoiceProgress`/`fieldChoiceInfo` got real `useMemo` (they're plain derived data, not delegate-safe), and the hook's whole return value is now `useMemo`'d too, so `selection` itself (the single prop `ActionBar` receives) is reference-stable when nothing in it actually changed.
- **`PlayerBoardPanel`'s prop contract changed**, not just wrapped in `memo()`: `onCardTap`/`onAttachedDonLabelTap` now take `ownerPlayerId` as a parameter (supplied internally from `board.playerId`, which the component already receives) instead of MatchScreen pre-binding it into a new closure per side per render; same for `canGiveDonOnCard`/`onGiveDon` taking `board` internally instead of a pre-bound `topPlayerBoard`/`bottomPlayerBoard` closure. This means `MatchScreen` can pass `selection.handleCardTap`/`selection.canGiveDonOnCard`/etc. straight through with **zero new closures**, sidestepping a much bigger problem: those props are assembled inside JSX that sits *after* several early returns in `MatchScreen`, and hooks (`useCallback`) can't be called after a conditional return (Rules of Hooks). Only one call site for `PlayerBoardPanel` exists in the whole repo (`PlayerSideRow`, itself only called twice, both in `MatchScreen`), so this was a contained, low-blast-radius change — confirmed by grep before touching it.
- `MatchScreen.tsx` itself: a new block of hooks (`useMemo`/`useCallback`) was hoisted to the top of the component, above every early return, computing null-safe versions of `bottomPlayerId`/`topPlayerId`/`actingPlayerId`/`bottomPlayerBoard`/`topPlayerBoard`/`battlePowerInstanceIds`/`openZoom`/the attack-target-hover handler/per-side `DockHand` callback bundles. A narrowing guard (`if (!bottomPlayerBoard || ...) return null;`) right after the existing `if (!matchState) return (...)` early return hands the rest of the function the same non-null names it already used, so the ~2,000 lines of JSX after that point needed only the specific prop-value edits described above, not a rewrite.
- `ActionLogPanel`/`ActionLogDock` were wrapped in `React.memo` too (not in the original 7) — their props (`log`, `playerNames`, `viewerPlayerId`) are already direct state reads, no extra wiring needed, so this was a free, low-risk win found while in the area.

**Deliberately deferred: `BoardCardTile`, `DonStack`, `PileStack`.** Their `card`/`cards` props come straight from `projectPlayerBoard`'s output, which rebuilds every `CardView` fresh on every *real* board change (no per-card memoization inside the projection layer itself). So even wrapped in `React.memo`, these three would get a new prop reference every time `PlayerBoardPanel` legitimately re-renders — comparison overhead, zero skip-render benefit. The actual fix (memoizing individual `CardView` construction inside `board/projection` so an unrelated card's props stay referentially stable) is a data-layer change, not a component-wrapping one — tracked as a **Phase 1.5 follow-up**, not done here.

Rollup's bundling step after that hung in this sandbox (same class of issue as Phase 0's jsdom hang — not investigated further, out of scope).

## Phase 2 results (pre-match asset preload — done, scoped to standard hotseat/CPU matches)

Implements the user's own proposed mechanism ("give a loading screen to copy all the required images before the match, cache them, so there's no more calling/recalling mid-match"), corrected during design from literal `localStorage` (wrong tool — synchronous, string-only, ~5-10MB quota) to the browser's **Cache Storage API** (`caches.open()`/`match()`/`put()`), the same mechanism service workers use for offline assets.

- **`src/cards/assets/cacheStorageAssetManager.ts` (new)** — the first real implementation of the `AssetCacheManager` interface that `assetCache.ts` had documented as a Phase-2 seam since Phase 0. `get()` returns a `blob:` object URL (via `Response.blob()` + `URL.createObjectURL()`), not the cache's own URL — that substitution is what actually guarantees zero network involvement mid-match, regardless of what cache-control headers the asset host sends. Cache key = the asset's own remote URL (not `cardDefinitionId` — that's normalized to the shared card *number*, not the printing, per `savedDeckToSetupInput.ts`'s existing doc comment — and not `card_image_id`, which is a different code path's key for the card-library/deck-builder browsing flow). Degrades to safe no-ops (never throws) everywhere `caches` is unavailable — old/locked-down browsers, private-browsing modes, and this project's own Node-environment test runs all exercise that branch.
- **`src/app/lib/matchAssetPreload.ts` (new)** — `preloadMatchAssets(images, cacheManager, onProgress)` orchestration: dedupes by distinct URL (leader art especially is frequently shared across the read deck), fetches/caches each with `Promise.all`, reports `{loaded, total}` progress per distinct URL resolved, and — critically — never lets one broken/slow asset host abort the whole batch: a failed URL just falls back to the original remote URL for that one entry (same as today's un-preloaded behavior), never dropped to a broken `<img>`. Also exports `revokeMatchAssetBlobUrls()`, since `MatchScreen` never actually unmounts on navigation (`if (!isMatchScreen) return null`) — component-unmount cleanup would never fire, so blob URLs from the *previous* match must be revoked explicitly at the start of each new preload run instead.
- **`matchStore.ts`**: added `applyCardImages(images)` — a small, separate action (`set({ cardImagesByDefinitionId: images })`) rather than a `startMatch()` parameter, since `startMatch()` stays synchronous and preloading is async; `MatchScreen` calls it once the preload promise resolves.
- **`MatchScreen.tsx`**: a new effect fires once per distinct `startMatch()` call (keyed on `startedWithDeckIds`'s object identity, which is fresh every real match start — see `matchStore.ts`), reads the just-started match's `cardImagesByDefinitionId`, runs `preloadMatchAssets`, then swaps the result in via `applyCardImages`. The existing `if (!matchState) { ...Starting match... }` gate was extended with a second gate — `if (current.screen === 'match' && !assetPreload.ready) { ...Loading card art, N / M... }` — so the board never mounts with any pending image fetches for a standard match. **Deliberately scoped to `current.screen === 'match'` only** (i.e. the existing `startMatch()`-driven hotseat/CPU path) — Online/Casual matches hydrate via `hydrateOnlineMatch()` (art streams in incrementally as opponents' cards reveal; blocking room-join on a full preload would be a regression) and Play Test's frequently-regenerated decks would fight their own churn. Both are explicitly out of scope for this pass, not overlooked.

**Backlogged, not implemented — "Phase 2.1":** a smaller/thumbnail board-tile image variant, distinct from the full-resolution zoom/preview asset. Raised as a natural follow-on to Phase 2 (a thumbnail would be even cheaper to preload) but explicitly deferred per the user's own call: a naively-downscaled thumbnail would look blurry blown back up on desktop's larger board tiles, and doing this properly needs a real asset-pipeline decision (build-time downscale vs. two hosted variants per printing), not a quick tweak. Tracked here so it isn't lost, not scheduled.

### Phase 2 production incident: blob: URLs, fixed to data: URLs (2026-07-19)

The very gap flagged above ("not verified by rendering... in production") turned out to be load-bearing: the user hit two real bugs live, both now fixed.

1. **Double-prefixed URL.** `cardImagesByDefinitionId` had always held root-relative paths; the original preload started storing `blob:` object URLs into that same map. `src/app/lib/assetUrl.ts`'s `resolveAssetUrl()` — the single seam every card render goes through — only recognized `http(s)://` as "already absolute," so it glued the production Vercel Blob CDN origin onto the front of every `blob:` URL, producing garbage like `<cdn-origin>/blob:https://one-piece-tcg-online.vercel.app/<uuid>`. Fixed by broadening the guard to detect any URI scheme (`/^[a-z][a-z0-9+.-]*:/i`), not just `http(s)`.
2. **`blob:` URLs are fundamentally the wrong tool here, not just mis-prefixed.** After fix #1, the network tab showed a clean, correctly-formed `blob:` URL — still failing to load. Root cause: `URL.createObjectURL()` results are only reliably valid for immediate, same-task use (or until explicitly revoked); this preload stores the resolved URL in Zustand state to be read by React an indeterminate time later — a card's `<img>` may not even attempt to load until it's drawn/played, and `CardImage.tsx` uses `loading="lazy"` on top of that, deferring the actual load further. That "resolve once now, consume much later" pattern is a known-fragile use of blob: URLs. **Fixed by switching `cacheStorageAssetManager.ts`'s `get()` to return a `data:` URL (base64-inlined bytes via `Blob.arrayBuffer()` + `btoa`, not `FileReader` — Node has no `FileReader` global, so a `FileReader`-based version could never be exercised by this project's Vitest suite the way the current implementation now is) instead of a `blob:` object URL.** `data:` URLs are inert, self-contained strings — no lifecycle, no revocation, safe to store and read whenever. This also let `MatchScreen.tsx` drop the `preloadedBlobUrlsRef`/revoke-on-new-preload plumbing entirely (nothing to revoke anymore); `revokeMatchAssetBlobUrls()` is kept in `matchAssetPreload.ts` as a defensive no-op/contract-tested utility rather than deleted, in case a future `AssetCacheManager` implementation ever does hand out real blob: URLs again.

**Why this wasn't caught beforehand:** this sandbox cannot run a real browser, so nothing here could exercise "create a blob: URL now, read it back after an async gap, through a lazy-loaded `<img>`" — the exact sequence that broke in production. The standalone-script verification approach used throughout Phase 2 (bypass Vitest, run the real module in plain Node/tsx) proved the *logic* was correct in isolation, but a same-process, same-tick smoke test can't reproduce a cross-time browser resource lifecycle bug — this is a real limit of what "verify in this sandbox" can mean, not something a better test in this sandbox would have caught. Real coverage for `cacheStorageAssetManager.ts`'s Cache-Storage-available path (`get()`/`put()`/`has()`/`evict()`, mocking `caches` + `fetch` in Node) was added after the fact — it exercises the exact `Blob -> data: URL` conversion now, but note it still can't reproduce the "read a stored URL much later, through `loading=\"lazy\"`" timing that was the actual root cause; only a real browser can.

**On "why does this look like it's still loading ad hoc mid-match" (a fair question, asked before the blob: URL bug was found and still true after the fix):** it is, by design of `CardImage.tsx`'s `loading="lazy"` — Cache Storage warming happens entirely upfront on the loading screen (all deck art is fetched once, before the match ever renders), but each `<img>` tag still defers actually decoding/painting until it's about to become visible (a card is drawn, played, or otherwise scrolled/revealed into view), which is what makes image loads appear staggered over the course of a match rather than all at once. With `data:` URLs there is no real network cost paid at that later point (a `data:` URI is inlined, not fetched) — so the staggering is now purely a decode/paint-timing artifact, not a network stall, which is what Phase 2 actually set out to eliminate. Making on-screen-at-match-start cards `eager` instead of `lazy` (so the very first paint has zero deferred decodes too) is a reasonable small follow-up, not done in this pass.

### Phase 2 production incident #3: preload fetched the wrong origin entirely (2026-07-19, same day)

After the `data:` URL fix, the user reported EVERY card blank (leader `<img>` confirmed via inspection to be rendering the "No image available" placeholder — `imageUrl` effectively unusable) and, decisively, opened the browser's Cache Storage panel: every single cached entry showed `Content-Type: text/html` and `Content-Length: 0`.

**Root cause:** `cardImagesByDefinitionId` values are root-relative paths (e.g. `/card-images/OP16/OP16-038_EN.webp`) — only `assetUrl.ts`'s `resolveAssetUrl()` knows to prepend the production Vercel Blob CDN origin (`VITE_ASSET_BASE_URL`) to that. `matchAssetPreload.ts`/`cacheStorageAssetManager.ts` were calling `fetch()` on the RAW stored path directly, never routing it through `resolveAssetUrl()` first — every single card render path (`CardImage.tsx`) does this correctly, but the new preload path was added without threading it through the same seam. In production, `fetch('/card-images/...')` resolves against the app's OWN origin, not the CDN — and `card-images/` is deliberately excluded from the app's own deployment bundle (`.vercelignore`), so every one of those requests silently "succeeded" (200 OK) against the SPA's fallback route, returning an empty HTML response, which then got base64-encoded and stored as a `data:text/html;base64,` "image" that could never render.

**Fix, three parts:**
1. `matchAssetPreload.ts` now calls `resolveAssetUrl(url)` before handing anything to `cacheManager.put()`/`.get()` — the cache key and the actual fetch target are both the resolved absolute URL; the result map still keys/falls back on the original raw path (unchanged shape for `applyCardImages()`/`CardImage.tsx`, which already resolves on its own).
2. `cacheStorageAssetManager.ts`'s `put()` now also rejects any response whose `Content-Type` doesn't start with `image/`, even if `response.ok` is true — a defensive guard against this exact "wrong origin, 200 OK, wrong body" failure mode ever silently recurring. A rejected asset falls back to the original URL for that one card, same as any other `put()` failure.
3. `CACHE_NAME` bumped `v1` -> `v2` so every browser that already has a Cache Storage full of bogus `text/html` entries from before this fix gets a clean slate, rather than those dead entries lingering unused.

**Verified** via `vite-node` (not `tsx` — `tsx` doesn't process `import.meta.env`, so it can't reproduce the real `VITE_ASSET_BASE_URL`-driven resolution at all) with the actual production CDN origin set as an env var: confirmed `fetch()` is now called with the fully-resolved CDN URL instead of the raw relative path, and the resulting `data:image/webp;base64,...` decodes back to the expected bytes. Also confirmed the content-type guard rejects a simulated wrong-origin/HTML response and falls back to the original URL instead of caching it. Full `vite build` still passes clean (595 modules).

**Pattern across all three Phase 2 incidents:** every one was a case of new preload code re-deriving or bypassing logic that a DIFFERENT, already-correct call site (`CardImage.tsx`/`resolveAssetUrl()`) already handled — first the CDN-prefix scheme guard, then blob: URL lifecycle, now skipping URL resolution entirely. The lesson (see [[project_phase2_match_asset_preload]] memory): any new consumer of `cardImagesByDefinitionId` values must go through the exact same resolution path existing renderers use, not assume the raw stored value is directly usable.

## Plan (in order — each phase should be measured before starting the next)

**Phase 0 — Baseline measurement.** Profile before touching anything: React DevTools Profiler render count per action, Chrome Performance panel with 4x CPU throttle, on a standard turn (attack + block + DON attach). Without this, "should be faster" isn't verifiable.

**Phase 1 — Cut the re-render blast radius (highest ROI, lowest risk).**
- `useMemo` the board projection calls, keyed on the state/images actually needed — stop rebuilding both players' full board on every render.
- Add `React.memo` to the leaf/mid components (`BoardCardTile`, `PlayerBoardPanel`, `DockHand`, `DonStack`, `PileStack`, `ActionLogPanel`, `ActionBar`) so a change to one card doesn't reconcile the whole board.
- These two have to land together — memoizing a component that's still handed a freshly-rebuilt projection object every render gets no benefit.
- Longer-term: `MatchScreen` itself is a 2,501-line monolith; consider narrowing what it reads from the store (per-zone/per-field selectors instead of the whole `state`) so it isn't the thing re-evaluating on every action in the first place.

**Phase 2 — Image weight and layout stability.**
- ~~A smaller board-tile image variant separate from the zoom/full-art asset~~ — reframed and done as **pre-match Cache Storage preloading** instead (see Phase 2 results above): every deck's art is fetched once into Cache Storage behind a loading screen before the board mounts, so mid-match rendering reads local `blob:` URLs with zero network involvement, regardless of image weight. A genuinely *smaller* thumbnail variant is still backlogged as **Phase 2.1** (see above) — that's a real asset-pipeline decision, not done here.
- Still open, not done in this pass: explicit `width`/`height` or `aspect-ratio` on every card `<img>` (layout-shift risk), `loading="lazy"` / `decoding="async"` for off-screen or face-down zones.

**Phase 3 — Animation/interaction audit under throttling.** Re-verify (empirically, not just by reading code) that no rAF loop or `getBoundingClientRect` polling keeps running after an animation finishes, under actual CPU throttling rather than an idle desktop.

**Phase 4 — Verification.** Re-run Phase 0's profiling after each phase and diff render counts / main-thread time. Consider a lightweight render-count regression test so memoization doesn't silently rot later.

## Known limitations
- Can't get real FPS/main-thread numbers from this environment — Phase 0's commit-count harness needs to run locally or in CI (this sandbox's jsdom hangs on any test, unrelated to this project's code) or on an actual device/throttled browser profile.
- A genuinely smaller thumbnail board-tile image variant (as opposed to Phase 2's "cache the existing full-size art" preload) implies either a build-time image pipeline or a second generated asset per printing across a 906MB / 8,346-file image set, and would look blurry on desktop if done naively (upscaling a small thumbnail) — backlogged as **Phase 2.1**, a real scoping decision, not a quick fix.
- Memoization only helps where props are reference-stable; if a parent keeps passing new inline objects/callbacks into a memoized child, the memo does nothing. Phase 1 needs to check callback/prop identity, not just wrap components in `memo()`.
- **Phase 1 is unverified by both the TypeScript compiler and any rendered/manual test in this sandbox** (see Phase 1 results above) — treat it as needing a local `tsc` run + a manual playtest before merging, not as done-and-safe.
- **Phase 2 is unverified by the TypeScript compiler** (same sandbox `tsc` limitation as Phase 1, now confirmed to be memory/CPU-bound on `effect-v2-output/`'s generated files rather than an I/O hang — see Phase 2 results above) **and by rendering/manual playtest** — `vite build`'s full bundle succeeding and the Vitest suites' logic being independently confirmed via standalone scripts are real signals, but neither is a substitute for `tsc` or seeing the loading screen actually work on a real match start. **Concretely proven by the production incident** (see above): this exact gap is what let a real bug reach production twice before being caught.
- Phase 1.5 (not started): per-card memoization inside `board/projection` so `BoardCardTile`/`DonStack`/`PileStack` can actually benefit from `React.memo` — currently skipped because their `card`/`cards` props are rebuilt fresh by the projection on every real board change.
- Phase 2's preload is scoped to the standard hotseat/CPU `startMatch()` path only — Online/Casual matches (`hydrateOnlineMatch`) and Play Test (`startPlayTest`) do not get a preload gate at all in this pass; see Phase 2 results above for why.
- This sandbox's Vitest runs showed intermittent hangs specific to `cacheStorageAssetManager.test.ts` (passed most attempts, hung on others, with identical file content) — flagged as sandbox flakiness rather than a code defect (see Phase 2 verification notes), but worth a clean local run to confirm before fully trusting it.
