## 2026-07-16 field-removal replacement + conditional-aura + attribute-filter pass

Investigated the four clusters flagged for this session (field-removal K.O. replacement, OP13 static removal immunity, conditional Blocker/cost auras, attribute-filter gaps) against the *current* `templateDefs.ts`/`factories.ts`/`interpreter.ts`/`koAttempt.ts`, not just the card comments (comments were sometimes stale — e.g. ST23-001/ST23-002 were already fully closed in code but still carried a `PARTIAL` comment, and OP12-070/OP12-061's field-removal clauses were already closed but the comment claimed otherwise). Closed **21 cards** using only existing primitives plus two small, mechanical plumbing additions (both used a pre-existing runtime code path that just wasn't exposed through the relevant TS param type):

1. **`registerKoReplacementSelf` action-kind widening** — `factories.ts`'s shared `koReplacementAction()` helper already implemented `trashSource`/`returnSourceToHand`/`restSource`/`giveSelfPowerPenalty`/`giveLeaderPowerPenalty`/`moveTargetToLifeFaceDown`/`trashSelfAndDraw` for K.O. replacements, but the `registerKoReplacementSelf` template-fn's TS signature only exposed a subset (these were previously aura-only in the type, even though `record.sourceInstanceId === appliesToInstanceId` for a self-registration, so they work identically). Added the missing fields to the type in `templateDefs.ts`. No interpreter/runtime change needed.
2. **`KoReplacementCharacterFilter.typeIncludes`** — the `restCharacter` K.O.-replacement action could filter by `minCost`/`excludeSourceName` but not by tribal type ("rest 1 of your {X} type Characters instead"). Added `typeIncludes` to the filter type (`game.ts`) and wired it into `eligibleRestCharacters` in `koAttempt.ts`, reusing the file's existing `typeIncludes()` helper.
3. **`attribute` filter wiring** — `SearchFilter.attribute` already existed and was used for deck search, but the `controllerCharacters`/`opponentCharacters` selectors (used by `addPower`, `setActiveControllerCharacter(s)`, etc.) silently dropped an `attribute` field even when present in the IR (a real "field exists elsewhere, not wired here" gap, exactly as flagged). Added `attribute?: Attribute` to `CharacterMoveFilter`, the `opponentCharacters` selector, `TargetFilter`, and `setActiveControllerCharacter(s)`'s filter type, then wired the actual filtering into both `controllerCharacters` and `opponentCharacters` cases in `interpreter.ts`.
4. **`addCostAuraControllerCharacters` + `anyOfNames`** — the sibling `addKeywordAuraControllerCharacters` already supported targeting an aura at "the source card itself" via `anyOfNames: ['OwnCardName']` (used e.g. on OP02-074), but the cost-aura equivalent silently dropped `anyOfNames` in its factory. Added it, mirroring the keyword-aura implementation exactly. This is what makes "if condition, this Character gains [Blocker] + N cost" static abilities expressible as a self-scoped aura pair.
5. **New gate `selfCharacterBaseCostCount`** — `selfCharacterCostCount` existed for *current* cost only; card text for ST25-002/ST25-005 explicitly says "base cost", so added the base-cost analog (mirrors the existing `selfHasCharacterBasePowerAtLeast` single-check gate, just as a counted version). Registered in `capability/registry.ts` too (capability:doc was failing to build without it).

**Explicitly NOT closed — do NOT copy these patterns blindly:**
- **OP04-082, OP11-110** — "rest your Leader or 1 [Named Stage/Leader] instead" — a *choice* between two different replacement-action kinds/targets. `KoReplacementAction` is a single discriminated action, not a chooseOne-style branch; no existing primitive composes two heterogeneous replacement actions into an either/or. Needs either a `chooseOne`-wrapped replacement action or a new composite action kind.
- **OP12-048** — "rest this Character AND trash 1 card from your hand instead" — a *compound* (both-parts-required) replacement action. Same root cause: `action: KoReplacementAction` is singular, not an array/sequence. Needs the K.O.-replacement action shape extended to accept a small sequence, analogous to how `payAbilityCosts` already accepts an array of `AbilityCost`.
- **OP12-061's `[Activate: Main]` DON!!−1 clause** — "next play of [Trafalgar Law] cost 4+ is reduced by 2" — `addNextPlayFromHandCostDiscount`'s `controllerCharactersInHand` aura group filters by `anyOfTypes` only (checked in `power.ts`'s `targetInAuraGroup`), not by card name, so a name-scoped one-shot next-play discount on a *different* card than the source isn't expressible without extending that group with `anyOfNames` support (small, but touches the runtime resolver, not just a param type — did not attempt this pass).
- **OP15-093's `<Slash>` attribute grant** — the `[Rush: Character]` keyword half of this card is now fixed (was wrongly mapped to plain `rush`, corrected to `canAttackCharactersWhileSummoningSick`), but *granting* an attribute to a target is a capability that does not exist anywhere in the engine (only *checking* an existing attribute does, via the fix in this pass). Needs a new continuous "add attribute" op — likely low ROI (1 known card).
